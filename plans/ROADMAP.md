# Sentinel — Build Roadmap

Current state: opaque tokens, no client registry, authorization code flow + PKCE works end-to-end.

Each phase below is independently shippable. Do them in order — each one builds on the previous.

---

## Phase 1 — Client Registry

**What's missing:** Sentinel accepts any `client_id` string. There's no list of registered clients. Anyone can point an app at Sentinel and it works. `redirect_uri` is not validated against anything.

**What this phase adds:**
- `Client` table in the DB
- Admin API to register/list/delete clients
- `client_id` generated by Sentinel (`clt_` + random hex)
- `client_secret` generated for confidential clients (optional, hashed in DB)
- `redirect_uris` stored per client, validated on every `/authorize` request
- `allowed_scopes` per client
- `AuthCode` and `Token` foreign key to `Client` instead of bare string

**After this phase:** Sentinel properly rejects unknown clients and mismatched redirect URIs. The auth server has real authority over who can use it.

See: [`01-client-registry.md`](./01-client-registry.md)

---

## Phase 2 — JWT Access Tokens

**What's missing:** All tokens are opaque random hex. Every API that wants to validate a token must call `/introspect`. No JWKS. No local validation.

**What this phase adds:**
- RSA key pair generated on startup (or loaded from env/file)
- `GET /.well-known/jwks.json` exposing the public key
- Access tokens are now signed JWTs (RS256) containing `sub`, `email`, `scope`, `client_id`, `iss`, `iat`, `exp`
- Token format is per-client (stored in `Client` table: `tokenFormat: 'jwt' | 'opaque'`)
- `/introspect` still works for opaque clients; JWT clients validate locally via JWKS
- `kid` in JWT header links to the correct JWKS key

**After this phase:** Resource servers can validate tokens without calling Sentinel. The auth server is no longer a dependency on every API request.

See: [`02-jwt-tokens.md`](./02-jwt-tokens.md)

---

## Phase 3 — Refresh Tokens

**What's missing:** Access tokens have a 24-hour lifetime — a workaround for the absence of refresh tokens. This is too long for JWTs (can't revoke easily) and unnecessary once refresh tokens exist.

**What this phase adds:**
- `RefreshToken` table in DB (opaque, long-lived, stored hashed)
- `/oauth/token` returns both `access_token` (15 min) and `refresh_token` (30 days)
- `POST /oauth/token` with `grant_type=refresh_token` issues a new access token
- Refresh token rotation: old token invalidated on use, new one issued
- Access token lifetime dropped to 15 minutes
- `POST /oauth/revoke` endpoint to invalidate a refresh token (logout)

**After this phase:** Users stay logged in without 24-hour access tokens sitting around. Logout actually works. Token lifetime is sane.

See: [`03-refresh-tokens.md`](./03-refresh-tokens.md)

---

## Phase 4 — OIDC Compliance

**What's missing:** Sentinel accepts `scope=openid` but doesn't issue an `id_token`. No `/userinfo`. No discovery endpoint. Not technically OIDC-compliant.

**What this phase adds:**
- `id_token` issued alongside `access_token` when `scope=openid` (always a JWT)
- Standard OIDC claims in `id_token`: `sub`, `email`, `email_verified`, `iat`, `exp`, `iss`, `aud`
- `GET /userinfo` endpoint (returns claims based on granted scopes)
- `GET /.well-known/openid-configuration` discovery document
- `scope=profile` support: adds `name`, `given_name` claims (requires adding these fields to `User`)

**After this phase:** Any standard OIDC library (Auth.js, openid-client, etc.) can integrate with Sentinel using just the issuer URL.

See: [`04-oidc-compliance.md`](./04-oidc-compliance.md)

---

## Phase 5 — Multi-Tenancy (Realms)

**What's missing:** One user pool. If you want to offer Sentinel as a service to multiple teams or companies, their users and clients need to be isolated.

**What this phase adds:**
- `Tenant` table (slug, name, settings)
- `User` and `Client` scoped to a `Tenant`
- Tenant-scoped URLs: `/:tenantSlug/oauth/authorize` etc.
- Tenant admin API (create tenant, manage users/clients within it)
- Super-admin API (manage tenants)

**Complexity note:** This is the biggest architectural change. Everything needs to be scoped. Do phases 1–4 first — this is only worth it if the core auth works correctly.

See: [`05-multi-tenancy.md`](./05-multi-tenancy.md)

---

## Phase 6 — Admin UI

**What this adds:**
- Web UI to register clients (instead of raw API calls)
- View registered clients, active tokens, users
- Per-client config: token lifetime, token format, allowed scopes, PKCE required
- Refresh token session management (see active sessions, revoke them)
- Key rotation UI (generate new RSA key pair, phase out old one)

**Tech choices:**
- Could be a separate Next.js app that calls Sentinel's admin API
- Could be a built-in Express static serve + simple HTML (simpler, like Keycloak's older UI)
- Keep it separate from the auth server — admin UI is a different concern

---

## Where to Start

**Start with Phase 1.** Client registry is the foundational correctness problem. Without it, Sentinel is a prototype. With it, it's a real auth server. Everything else is additive.

The first thing to build in Phase 1: the `Client` model + migration, then the admin endpoint to register a client, then wire up the validation in `/authorize`.
