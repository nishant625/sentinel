# 11 — Client Registry, Tenancy, and Schema Design

Three questions answered here:
1. Should client IDs be generated by the auth server?
2. Should clients live in a separate table from users?
3. Auth0 model vs Keycloak model — what's the actual difference?

---

## Should the Auth Server Generate Client IDs?

Yes. Always.

Right now Sentinel accepts any string as `client_id` — it never checks whether that client is registered. This means anyone can point a random app at Sentinel and it will happily process their auth requests. That's fine for learning but it's not how a real auth server works.

In a real system, the flow is:

```
Developer creates a client via admin API/UI
  → auth server generates a client_id (e.g. "clt_a7f3b2c9d4e5")
  → auth server optionally generates a client_secret (for confidential clients)
  → developer puts client_id in their app
  → developer registers redirect_uris
  → auth server now validates all of these on every authorize request
```

The auth server owns the registry. It issues the IDs. This is what gives the auth server the authority to reject unknown clients.

### What a Generated Client ID Looks Like

Different servers use different formats:

| Server | Format example |
|---|---|
| Auth0 | `DGv6Ra7Nj6LK8XA5pVkGNRg2pvd7LX3H` (random alphanumeric) |
| Keycloak | `my-react-app` (human-chosen name, like a username) |
| Okta | `0oa1b2c3d4e5f6g7h8i9` (prefixed random string) |
| Google | `123456789-abc.apps.googleusercontent.com` |

Keycloak lets developers pick their own client ID (just has to be unique). Auth0 generates it. Sentinel should probably generate it — using `crypto.randomUUID()` or a prefixed random string like `clt_` + random hex. Cleaner, no collisions, no need for uniqueness validation on the developer's side.

## Should Clients Be Separate From Users?

Yes. They're completely different things.

| | User | Client |
|---|---|---|
| What it is | A human who logs in | An application that requests tokens |
| Who creates it | Signs up / admin seeds | Developer registers via admin |
| Authenticates with | Email + password | client_secret (or PKCE for public clients) |
| Identified by | email / user ID | client_id |
| Has sessions | Yes | No |
| Has tokens | Access tokens issued on their behalf | Optionally client_credentials tokens |

In Sentinel's current schema, `clientId` in `AuthCode` and `Token` is just a `String`. It's not a foreign key to anything. Sentinel has no `Client` table. That's what needs to change.

### What the Client Model Looks Like

```prisma
model Client {
  id           Int      @id @default(autoincrement())
  clientId     String   @unique   // "clt_a7f3b2c9d4e5" — generated
  clientSecret String?            // null for public clients (SPAs, mobile)
  name         String             // "My React App" — human label
  isPublic     Boolean  @default(true)
  redirectUris String[]           // allowed redirect URIs
  scopes       String   @default("openid")
  createdAt    DateTime @default(now())

  authCodes AuthCode[]
  tokens    Token[]
}
```

Then `AuthCode.clientId` and `Token.clientId` become foreign keys to `Client.clientId` instead of bare strings. Every authorize request now validates the `client_id` exists and the `redirect_uri` is in the registered list. Unknown clients get rejected at the gate.

## Auth0 Model vs Keycloak Model

This is the tenancy question. It comes down to: **who are your users and how isolated do they need to be?**

### Single-Realm (No Tenant Layer)

```
Sentinel
├── Users (everyone in one pool)
│   ├── alice@acme.com
│   ├── bob@startup.com
│   └── carol@personal.com
└── Clients
    ├── my-react-app
    ├── my-mobile-app
    └── my-admin-panel
```

One database. One user pool. Multiple clients (applications), all sharing the same users.

This is the right model when Sentinel is the auth server for **your own product**. All your users are in one place. Your web app, mobile app, and internal tools are all clients registered in the same Sentinel instance.

This is what Sentinel is right now (minus the Client table).

### Multi-Tenant / Multi-Realm

```
Sentinel
├── Tenant: Acme Corp
│   ├── Users: alice@acme.com, bob@acme.com
│   └── Clients: acme-web-app, acme-mobile
│
├── Tenant: Startup Inc
│   ├── Users: carol@startup.com, dave@startup.com
│   └── Clients: startup-app
│
└── Tenant: Personal Project
    ├── Users: eve@personal.com
    └── Clients: my-side-project
```

Multiple completely isolated environments inside one Sentinel instance. Users from Acme Corp cannot log in to Startup Inc's apps. Each tenant manages their own clients independently.

**This is what Keycloak calls "Realms."** When you run Keycloak, you create realms for each isolated customer or project. The Keycloak admin (you) manages the platform. Each realm admin manages their own users and clients.

**This is what Auth0 calls "Tenants."** Auth0 is a SaaS product — you sign up, you get a tenant. Your tenant has its users and apps. Another company signs up, they get their tenant. Auth0 is the operator of the platform. You are the tenant admin.

### The Schema Difference

Without tenants — Client and User are top-level:

```prisma
model User    { ... }
model Client  { ... }
model AuthCode { userId, clientId, ... }
```

With tenants — everything hangs off a Tenant:

```prisma
model Tenant {
  id      Int    @id @default(autoincrement())
  slug    String @unique   // "acme-corp" — URL-safe identifier
  name    String
  users   User[]
  clients Client[]
}

model User   { tenantId Int; tenant Tenant @relation(...); ... }
model Client { tenantId Int; tenant Tenant @relation(...); ... }
```

Now alice from Acme and alice from Startup are different rows with different `tenantId`. A client registered for Acme cannot be used by Startup's users.

The authorize URL also changes — it needs to know which tenant:

```
# Without tenants
GET /oauth/authorize?client_id=my-app&...

# With tenants (Keycloak style)
GET /realms/acme-corp/oauth/authorize?client_id=my-app&...

# With tenants (subdomain style, Auth0 style)
GET https://acme-corp.auth.yourservice.com/oauth/authorize?...
```

## What Should Sentinel Be?

If you're building this as a learning project or for one product — **start without tenants**. Add the Client table, validate client_id, generate IDs. That's already a massive step up from where it is now.

If you want it to be a general-purpose self-hosted auth server people can run (like a lite Keycloak) — **tenants are necessary but they're a significant complexity jump**. Add them after the core works: JWT, refresh tokens, client registry, OIDC compliance.

The sequence that makes sense:

```
Now:        Opaque tokens, no client registry, no PKCE state validation
Phase 1:    Client registry (validate client_id + redirect_uri against DB)
Phase 2:    JWT access tokens + JWKS
Phase 3:    Refresh tokens
Phase 4:    OIDC compliance (id_token, /userinfo, discovery endpoint)
Phase 5:    Multi-tenancy (realms)
Phase 6:    Admin UI
```

Each phase is independently useful and shippable. Detailed build plans are in `/plans/`.
